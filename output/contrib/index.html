collaboration model

The S4 project adopts the fork+pull model using git. This means developers work on forks of the S4 repo (i.e. their own clones of it). The S4 Team then merges these changes into the main S4 repositories when requested to do so by other developers.

GitHub has a good description of this model here.

version control

S4 is hosted on GitHub.

If you are new to git, gitref.org has a good tutorial.

submission guidelines

Before you start, send a message on the S4 mailing list, or create an issue in github, describing the proposed changes.

If you are contributing a substantial change to S4, we need you (or the company you work for) to sign our Yahoo! Open Source Contributor License Agreement.

Alternatively, you may complete a PDF version of the form and email it to one of the maintainers affiliated with Yahoo!

modifying the source code

First fork the repository containing the code you want to change. Then clone this repository in your development environment.

Make changes to the code, following these conventions:

  1. All public classes, interfaces, and methods should have informative Javadoc comments. Avoid trivial comments.
    • BAD: getCondensedLength(): Gets condensed length.
    • BETTER: getCondensedLength(): Number of characters in the string, excluding non alpha-numeric characters.
  2. Code Formatting
    • New Java source files should be formatted according to Sun's conventions with these exceptions:
      • Indent four spaces per level
      • Use spaces for indentation, not tabs. Tabs tend to break formatting in other editors. Instructions for doing this in eclipse can be found here.
    • If you are editing an existing file, mimic the conventions used in that file.
  3. Don't modify formatting, indentation, etc of lines of code that are not part of the change that you are making. This makes code diffs hard to understand.
    • If you must change formatting in a file, do that as a separate commit.

submitting changes

Once you are done with changes in your clone of the repository, push them to your forked repository.

Then, send us a pull request. This lets the rest of us know that you are done with your changes. Other developers may have comments, based on which you may make more changes to your fork.

After this, the S4 Team will merge your changes into the main S4 repository.

Thank you for making S4 better!
